# poc — reference integration

**Not a shipped library.** This directory is reference TypeScript code
showing how to load and call the spackle wasm module from a JS host.
Consumers building their own integration should lift and adapt —
it's not versioned, not packaged.

For the wasm architecture, the `SpackleFs` contract, and the decision
log, see [`../WASM.md`](../WASM.md) and [`../SUMMARY.md`](../SUMMARY.md).

---

## Runtime support

| Runtime | Version tested | Status |
|---|---|---|
| Node | 22.22.2 | ✓ verified (`just test-poc`) |
| Bun | 1.2.8 | ✓ verified (`just test-poc` runs via `bun test`) |

`wasm-pack build --target nodejs` emits a module that instantiates
eagerly at import time — no `await init()` boilerplate. Works under
both Node and Bun since both resolve CommonJS + ESM from the same
output.

Older runtime versions may work; they aren't exercised by CI.

## Quickstart — adapting for your host

Two reference adapters ship in `src/host/`:

- **`DiskFs`** — backs a `SpackleFs` with the local filesystem,
  rooted at a workspace directory. Enforces containment via
  canonicalize-then-check. Use this for server-side generation where
  outputs are real files.
- **`MemoryFs`** — backs a `SpackleFs` with an in-memory Map.
  Useful for preview flows, tests, and any case where you want to
  inspect generated output without touching disk.

Typical server usage:

```ts
import { DiskFs, check, generate, validateSlotData } from "./src/spackle";

const fs = new DiskFs({ workspaceRoot: "/var/spackle/workspace" });

// Validate a project's structure.
const checkResult = await check("/var/spackle/workspace/my-template", fs);

// Validate submitted slot data.
const slotResult = await validateSlotData(
    "/var/spackle/workspace/my-template",
    { project_name: "hello" },
    fs,
);

// Generate output.
const genResult = await generate(
    "/var/spackle/workspace/my-template",
    "/var/spackle/workspace/generated/abc-123",
    { project_name: "hello" },
    fs,
);
```

Preview flow (no disk writes):

```ts
import { MemoryFs, generate } from "./src/spackle";

const mem = new MemoryFs();
// Seed the project files into memory (from git, S3, wherever).
mem.insertFile("/project/spackle.toml", tomlBytes);
mem.insertFile("/project/README.md.j2", templateBytes);

const res = await generate("/project", "/output", slotData, mem);
// Inspect results:
const { files } = mem.snapshot();
// { "/output/README.md": ..., ... }
```

### Writing your own adapter

Implement the `SpackleFs` interface from `src/host/spackle-fs.ts`. The
shape is documented in `../WASM.md` but the load-bearing rules are:

1. **Sync only.** Every method returns a value directly (no Promise).
2. **Throw typed errors.** Use `fsError(kind, message)` to construct
   throwables with the `{ kind, message }` shape Rust decodes. `kind`
   must be one of the seven documented values; anything else maps to
   `"other"` and loses type info.
3. **Paths are absolute.** Reject relative paths with `invalid-path`.
4. **Containment is your job.** Nothing in the wasm or the spackle
   core enforces a workspace boundary — your adapter decides what
   "inside" means.

## Known caveats

1. **Hooks aren't supported yet.** `generate(..., { runHooks: true })`
   returns `{ ok: false, error: "hooks are unsupported in this
   milestone..." }`. The hooks path will come back via a separate
   `SpackleHooks` adapter; see `../WASM.md` for the design sketch.
2. **`Path::canonicalize` is gone from the Rust lib.** Doesn't matter
   for DiskFs (which canonicalizes host-side) but if your adapter
   expected spackle to normalize paths, it doesn't. Pre-normalize
   before passing into `generate`.
3. **Path encoding is UTF-8 only.** `JsFs` converts Rust paths via
   `to_string_lossy`. Non-UTF8 filenames will be mangled.
4. **`slugify` appears in `pkg/spackle.d.ts`.** Incidental export from
   tera's `slug` dep. Not part of our contract; ignore it.
5. **DiskFs TOCTOU.** Between DiskFs's containment check and the
   actual fs op, a symlink could be swapped. Userspace JS can't fully
   close this; production deployments accepting untrusted templates
   should layer OS-level sandboxing (Landlock on Linux, App Sandbox
   on macOS, a container).
